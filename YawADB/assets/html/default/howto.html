HOW TO

Why YawADB?

Despite a variety of similar applications, we decided to develop another one to better suit our needs. Since all appropriate names have been used, we couldn't come up with anything better than 'Yet Another Wireless ADB application', or YawADB for short. In expectation that this application can be useful for someone else we release it under GPL. The source code can be obtained from the project page: https://code.google.com/p/yawadb.

YawADB configures a rooted Android device for wireless access through Android Debug Bridge (ADB) without using USB cable connection. It can be launched either as an app (backward compatible up to API 1), or as a widget (compatible up to API 3, the first release featuring applications widgets).

YawADB is designed to be self aware and keep the information current. It can be configured to periodically re-check the status, but even if the refresh is disabled or infrequent, the application still tries its best to provide up-to-time feedback.

The application provides a number of customised features for someone who likes experimenting and know what they are doing.

Being not authorised to translate GPL text to other languages, we for consistency decided to leave the other text untranslated too. SVP, entschuldigen Sie-nous per queste molestias. Another reason for such a decision is that a widget must be installed in device internal memory where the size is often crucial. Since the text is not hard-coded, support for another language can be easily added.

Customised features

Port number. Specifies port to use for connection. Use default value 5555, unless it clashes with another application.

Auto-refresh (widget mode only). If auto-refresh is enabled, the widget re-checks the status of ADB daemon in the specified interval of time. If disabled (default value), the status is re-checked only on connectivity change (such as lost/regained connection, or airplane mode), when YawADB icon is tapped, or after enable/disable wireless connection using YawADB. Auto-refresh is recommended when there are several applications changing wireless ADB status, or ADB daemon is unstable on your device (a rather unlikely possibility).

SU path. The path of su program (e.g. /system/bin/su). If the full path is not specified, the directory should be listed in PATH environment variable.

If your device is rooted only for ADB, you may still succeed in running YawADB. For example, you may create a suid-rooted copy of /system/bin/sh (please, don't ask to go into the details!) and set this copy as SU Path for YawADB. Be aware that this makes your device unsafe by allowing to execute rooted commands without any notification! For that reason using a rooted device is certainly a preferred option.

Network. Name of interface, used for wireless ADB. The standard name is wlan0, however some devices use tiwlan0 or even eth0. It the name is specified as wifi (default value), YawADB will get IP address from Android WiFi service (WifiManager class). The parameter opens a door for trying mobile connections (such as rmnet), and if you succeed, take our congratulations.

Due to a rather awkward Android design the default option works only with IPv4 network. If the interface name is specified explicitly, the application will use IPv4 network, or IPv6 network, if IPv4 network is unavailable. If :6 is appended to the network name (e.g wlan0:6), the IPv6 network has preference over IPv4. (Yes, ADB currently does not support IPv6, but it hopefully will in the future.)

ADBD restart. In order to accept the new mode, ADB daemon should be restarted. In normal mode (default) the daemon is stopped using stop adbd command, which may fail, if the daemon is busy or stalled. In forced mode the daemon is stopped by sending SIGKILL to its PID. It almost guarantees the daemon to stop, but precludes it from performing clean-up. Use it with care.

Troubleshooting

YawADB icon is green, but ADB client does not see the device.

Assuming the client deviceget is connected to the network and not behind a firewall, re-entering  adb connect  will do the trick.

Wireless ADB is turned off, but ADB client does not recognise the device, connected to USB.

Disconnect the device and re-connect it. Alternatively you can restart the ADB server on the client side using

  adb kill-server
  adb start-server
